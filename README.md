## Q1. Разница между указателями и ссылками

| | Указатель | Ссылка |
|---|---|---|
| Может быть `nullptr` | ✅ | ❌ |
| Перепривязка | ✅ | ❌ |
| Инициализация обязательна | ❌ | ✅ |
| `sizeof` | 8 байт (64-бит) | не определён |
| `&x` | адрес самого указателя | адрес исходного объекта |

Ссылка — псевдоним объекта. Указатель — адрес в памяти, связан с ячейкой, а не объектом.

## Q2. Конструктор по умолчанию, типы конструкторов

**Конструктор по умолчанию** — вызывается без аргументов (`T()`). Генерируется автоматически если не объявлен ни один другой конструктор.

**Типы конструкторов:**
- **Default** `T()` — без аргументов
- **Converting/параметрический** `T(int)` — один или более аргументов
- **Copy** `T(const T&)` — инициализация из lvalue
- **Move** `T(T&&)` — инициализация из rvalue
- **Делегирующий** — вызывает другой конструктор того же класса через список инициализации

Автогенерация default ctor **подавляется** если объявлен любой другой конструктор (кроме copy/move).

## Q3. Типы размещения

- **Automatic (стек)** — локальные переменные. Создаются при входе в блок, уничтожаются при выходе.
- **Static** — глобальные, `static`-переменные. Живут всё время программы. Инициализация при первом использовании (для локальных static) или до `main` (для глобальных).
- **Dynamic (куча)** — `new`/`delete`. Время жизни контролируется программистом.
- **Thread-local** (`thread_local`) — по экземпляру на поток.

---

## Q4. Порядок инициализации объектов со статическим размещением

1. Объекты из одной единицы трансляции — в порядке объявления.
2. Между единицами трансляции — **порядок не определён** (static initialization order fiasco).
3. Константные выражения инициализируются до остальных (constant initialization).

**Решение проблемы**: использовать функции-синглтоны (Meyers' singleton):
```cpp
Foo& get_foo() { static Foo foo; return foo; }
```

## Q5. Что такое объект в C++

Объект — регион хранения данных с типом и временем жизни. Создаётся конструктором, уничтожается деструктором. Не всякое место в памяти — объект (до завершения конструктора объекта нет).

---

## Q6. Этапы жизни объекта

1. **Выделение памяти** (allocation).
2. **Инициализация** — вызов конструктора. С этого момента объект существует.
3. **Использование**.
4. **Завершение** — вызов деструктора. Объект перестаёт существовать.
5. **Освобождение памяти** (deallocation).

Нельзя использовать объект до конструктора или после деструктора — UB.

## Q7. `const` и его использование

```cpp
const int x = 5;           // нельзя изменить
const int* p;              // указатель на const int (объект нельзя менять)
int* const p = &x;         // const указатель (адрес нельзя менять)
const int* const p = &x;   // оба const
```

**Читать справа налево:** `int* const p` = "p — const указатель на int".

**`const`-метод** — `this` становится `const T*`, нельзя менять поля (кроме `mutable`):
```cpp
int get() const { return n; }
```

**`mutable`** — поле можно изменять даже в `const`-методе (для кэшей, счётчиков, мьютексов).

## Q8. `static` и его использование

- **В функции**: переменная инициализируется один раз при первом вызове, живёт до конца программы.
- **В классе**: принадлежит типу, а не объекту. Один экземпляр для всех объектов.
- **В файловой области**: внутреннее связывание (не видно из других единиц трансляции).
- **`static constexpr` в классе**: неявно `inline`, не требует отдельного определения.

---

## Q9. `virtual` и его использование *

- Виртуальный метод — вызывается по динамическому типу объекта (через vtable), а не по статическому.
- Виртуальный деструктор — обязателен, если класс будет удаляться через указатель на базовый класс.
- **Абстрактный метод**: `virtual void f() = 0;` — чисто виртуальный.

---

## Q10. `override` и его использование *

- Явно указывает, что метод переопределяет виртуальный. Ошибка компиляции если не переопределяет.
- Предотвращает ошибки при изменении сигнатуры в базовом классе.

```cpp
void f() override;  // ошибка если нет virtual f() в базе
```

---

## Q11. `template` и его использование *

- Шаблоны функций/классов — параметризация типами или константами.
- Инстанцируются при использовании (в момент компиляции).
- Определения шаблонов — только в header (неявно inline).

---

## Q12. `void` и его использование

- **Тип возврата**: функция ничего не возвращает.
- **`void*`**: безтиповый указатель, принимает любой указатель. Разыменовать нельзя. Обратное преобразование — только явным `static_cast`.
- **Параметр `void`**: `int f(void)` — архаичный C-стиль, эквивалентен `int f()`.

---

## Q13. `nullptr` и его использование

- Тип `std::nullptr_t`. Неявно приводится к нулевому указателю любого типа.
- **Не приводится к `int`** (в отличие от `NULL == 0`).
- Безопасен для перегрузки функций:
```cpp
void f(int);
void f(int*);
f(NULL);    // вызовет f(int) — неожиданно!
f(nullptr); // вызовет f(int*) — корректно
```

---

## Q14. `friend` и его использование

- **`friend`-функция**: имеет доступ к `private`/`protected` членам класса, но не является его методом.
- **`friend`-класс**: все методы этого класса имеют доступ к `private`.

```cpp
class C {
    friend void helper(C&);
    friend class X;
    int secret = 42;
};
void helper(C& c) { c.secret = 0; }  // OK
```

Не наследуется, не транзитивно.

---

## Q15. `explicit` и его использование

Без `explicit` — конструктор участвует в неявных преобразованиях.  
С `explicit` — только явный вызов.

```cpp
struct A { A(int); };
struct B { explicit B(int); };
void f(A); void g(B);
f(10);      // OK: неявное A(int)
g(10);      // ошибка: B(int) explicit
g(B(10));   // OK
```

**Правило**: ставить `explicit` на все однопараметрические конструкторы по умолчанию.

---

## Q16. `inline` и его использование

- **Для функций**: говорит линкеру, что функция может быть определена в нескольких единицах трансляции (не значит "встроить"). Инлайнинг — решение компилятора.
- **Для переменных (C++17)**: `inline int x = 0;` — можно в header без ODR-нарушений.
- `constexpr`/`consteval` функции — неявно `inline`. `static constexpr` члены — неявно `inline`.

## Q17. Какие методы генерируются компилятором автоматически

1. **Default constructor** `T()`
2. **Copy constructor** `T(const T&)`
3. **Move constructor** `T(T&&)`
4. **Destructor** `~T()`
5. **Copy assignment** `T& operator=(const T&)`
6. **Move assignment** `T& operator=(T&&)`

**Правило нуля**: если класс не управляет ресурсами — не объявляй ни одного специального метода.

**Правило пяти**: объявил один — явно определи все пять.

| Объявил | Default | Copy ctor | Copy op= | Move ctor | Move op= |
|---|---|---|---|---|---|
| Деструктор | ✓ | ✓ (depr) | ✓ (depr) | ✗ | ✗ |
| Copy ctor | ✓ | — | ✓ (depr) | ✗ | ✗ |
| Move ctor | ✗ | ✗ (del) | ✗ (del) | — | ✗ |

## Q18. Виды передачи параметров

```cpp
void f1(S s);          // по значению: копия
void f2(const S& rs);  // const-ссылка: без копии, без nullptr
void f3(S& rs);        // по ссылке: можно изменять
void f4(const S* ps);  // по указателю: можно nullptr
void f5(S&& rs);       // rvalue-ссылка: move-семантика
```

- Маленькие типы (`int`, `char`) — по значению.
- Большие только для чтения — `const T&`.
- Нужен null — указатель.
- Нужно изменить — `T&`.
- Move — `T&&` или по значению + `std::move`.

## Q19. Как передать строковый параметр

```cpp
void f(const char* s);         // плохо: нет длины, нет сигнала о владении
void f(const std::string& s);  // OK, но нельзя без конверсии строковый литерал → создаёт временный
void f(std::string_view sv);   // предпочтительно: принимает всё, без копии
```

`std::string_view` — пара (указатель, размер). Не владеет памятью. Опасность: данные должны пережить `string_view`.

## Q20. Как вернуть результат(ы) из функции

1. **Return value** (предпочтительно): `T f();`
2. **Через ссылку-параметр**: `void f(T& out);`
3. **Через указатель-параметр**: `void f(T* out);`
4. **`std::tuple`/`std::pair`** + structured bindings: `auto [a, b] = f();`
5. **Через поле структуры**: `Result f();` — самодокументируемо.

## Q21. Порядок вычисления аргументов функции

**Не определён** (unspecified). Компилятор волен вычислять аргументы в любом порядке.

```cpp
f(x++, x++);   // UB: оба изменяют x без sequenced-before
```

## Q22. Пост- и пре- инкремент

- **`++x`** (пре-): инкрементирует, возвращает ссылку на изменённый объект. Для пользовательских типов: нет лишних копий.
- **`x++`** (пост-): создаёт копию старого значения, инкрементирует, возвращает копию. Дороже для не-trivial типов.

```cpp
C& operator++() { ++val; return *this; }         // пре-
C operator++(int) { C tmp = *this; ++val; return tmp; } // пост-
```

---

## Q23. Абстрактный класс *

- Имеет хотя бы один чисто виртуальный метод (`= 0`).
- Нельзя создать экземпляр напрямую.
- Может иметь конструктор (для инициализации в наследниках).
- Наследник обязан переопределить все чисто виртуальные методы, иначе тоже абстрактный.

---

## Q24. Тернарный оператор

`condition ? expr_true : expr_false` — это **выражение**, а не оператор ветвления.  
Разница: у if/else результат — statement (нельзя использовать как значение).  
Тернарный — expression, возвращает значение, типы обеих веток должны совпадать (или одна конвертируема в другую).

---

## Q25. `&` vs `&&`

- `&` — **lvalue-ссылка**: связывается с lvalue (именованными объектами).
- `&&` — **rvalue-ссылка**: связывается с xvalue/prvalue (временными объектами).
- `const T&` — связывается и с lvalue, и с rvalue (продлевает время жизни временного).

Используется в move-семантике: `T&&` в конструкторе перемещения говорит "этот объект можно забрать".

---

## Q26. `=` vs `==`

- `=` — оператор присваивания (или инициализации при объявлении).
- `==` — оператор сравнения, возвращает `bool`.

Распространённая ошибка: `if (x = 5)` — всегда истинно (x присваивается 5, затем проверяется 5).

---

## Q27. Перегрузка операторов: 2 подхода

**1. Метод класса** (лучше для `=`, `[]`, `()`, `->`, унарных):
```cpp
C& operator+=(const C& rhs) { ... return *this; }
```
Левый операнд — `this`.

**2. Свободная функция** (лучше для бинарных коммутативных, `<<`, `>>`):
```cpp
C operator+(C lhs, const C& rhs) { lhs += rhs; return lhs; }
```
Оба операнда явные, можно симметрично конвертировать левый.

**Что нельзя перегрузить**: `::`, `.`, `.*`, `?:`, `sizeof`, `typeid`.

---

## Q28. Определение vs объявление

- **Объявление**: говорит компилятору о существовании сущности и её типе. Не резервирует память.
- **Определение**: выделяет память / предоставляет тело. Каждая сущность — ровно одно определение (ODR).

```cpp
extern int x;       // объявление
int x = 5;          // определение

void f(int);        // объявление функции
void f(int x) {}    // определение функции

class C;            // объявление (forward declaration)
class C { int n; }; // определение
```

## Q29. Move-семантика: `std::move`

`std::move(x)` = `static_cast<T&&>(x)` — приводит lvalue к xvalue.  
**Само по себе ничего не перемещает** — это делает move-конструктор/оператор получателя.

После `std::move(x)` объект `x` в валидном, но неопределённом состоянии.

**Зачем**: передать владение ресурсом без копирования:
```cpp
std::string a = "hello";
std::string b = std::move(a);  // b получает буфер a, a становится пустой
```

## Q30. Типы циклов в C++

```cpp
for (int i = 0; i < n; ++i) {}          // классический for
while (cond) {}                          // while: проверка до итерации
do {} while (cond);                      // do-while: проверка после итерации
for (auto& x : container) {}            // range-for: для итерируемых контейнеров
```

`do-while` — тело выполнится хотя бы один раз. `while` — может не выполниться ни разу.  
`range-for` — разворачивается в итераторный цикл `begin()`/`end()`.

## Q31. Зачем нужны `.h` файлы

Разделение **интерфейса** (объявления) и **реализации** (определения).

- `.h` подключается в несколько `.cpp` → каждый знает сигнатуры
- Реализация компилируется один раз (в `.cpp`)
- Позволяет скрыть детали реализации (библиотеки поставляются как header + `.so`)

**Правило**: в header — объявления, `inline`-функции, шаблоны, `constexpr`. В `.cpp` — всё остальное.

## Q32. Единица трансляции

Один `.cpp`-файл после обработки препроцессором (все `#include` подставлены, макросы раскрыты). Компилируется в один объектный файл.

## Q33. Объектный файл

Результат компиляции одной единицы трансляции. Содержит:
- Скомпилированный машинный код
- Таблицу символов (имена функций/переменных для линкера)
- Информацию о неразрешённых ссылках

Линкер объединяет несколько `.o` в исполняемый или библиотеку.

## Q34. Чем инициализируется `bool` по умолчанию

- **Глобальная/статическая `bool`**: `false` (zero-initialization).
- **Локальная `bool`**: **не определена** (garbage), если не инициализирована явно.

```cpp
bool b;        // локальная — неопределённое значение (UB при чтении)
static bool b; // false
bool b{};      // false (value-initialization)
```

---

## Q35. Варианты приведения типов по иерархии наследования

- **`static_cast<Derived*>(base_ptr)`** — downcast без проверки. UB если на самом деле не Derived.
- **`dynamic_cast<Derived*>(base_ptr)`** — с проверкой в runtime (требует виртуальных методов). Возвращает `nullptr` если не тот тип (для указателей) или бросает `std::bad_cast` (для ссылок).

**Upcast** (Derived* → Base*) — неявный, всегда безопасен.

|---|---|---|
| Область видимости | засоряет текущую | только `Enum::Value` |
| Неявное приведение к `int` | ✅ | ❌ |

| Базовый тип по умолчанию | не уже `int` | `int` |

## Q37. `struct` vs `class`

**Единственное отличие**: доступ по умолчанию.
- `struct`: `public` (поля и наследование)
- `class`: `private` (поля и наследование)

Технически эквивалентны.

---

## Q38. Спецификаторы доступа

- **`public`**: доступ всем.
- **`private`**: только членам класса и `friend`.
- **`protected`**: как private + доступ наследникам.

Разрешение перегрузки происходит **до** проверки доступа — лучшая перегрузка может быть private → ошибка.

---

## Q39. Конструктор и когда вызывается

Специальная функция-член без типа возврата, имя = имя класса. Вызывается при каждом создании объекта. Инициализирует поля через список инициализации (до тела).

---

## Q40. Деструктор и когда вызывается

`~ClassName()` — вызывается при уничтожении объекта: выход из области видимости, `delete`, конец программы (для статиков). Сначала тело деструктора, затем поля в обратном порядке объявления.

---

## Q41. Можно ли явно вызвать конструктор?

**Нет** (обычным способом). `obj.T()` — ошибка компиляции.  
Единственный способ явно вызвать конструктор — **placement new**: `new(ptr) T(args)`.

---

## Q42. Можно ли явно вызвать деструктор?

**Да**: `obj.~T()` — синтаксически корректно.  
Уместно **только** для объектов, созданных через placement new — чтобы уничтожить объект без освобождения памяти.  
Для обычных объектов — UB (двойное уничтожение).

---

## Q43. Оператор присваивания vs конструктор копирования

- **Copy constructor** `T(const T&)` — инициализация нового объекта: `T b = a;` или `T b(a);`
- **Copy `operator=`** — присваивание уже существующему: `b = a;`

```cpp
T b = a;   // copy constructor
T b;
b = a;     // copy operator=
```

---

## Q44. `new int(10)` vs `new int[10]`

- `new int(10)` — создаёт **один** `int` со значением `10`. Удалять через `delete`.
- `new int[10]` — создаёт **массив** из 10 `int`. Удалять через `delete[]`.

Путаница → UB.

---

## Q45. Можно ли `delete ptr` после `new T[5]`?

**Нет — UB.** `new[]` хранит количество элементов перед массивом. `delete[]` читает это число и вызывает столько деструкторов. `delete` не читает — вызывает только один деструктор, остальные утекают.

---

## Q46. Placement new

Конструирует объект на уже выделенной памяти, без новой аллокации:
```cpp
std::byte* buf = new std::byte[sizeof(S)];
S* ps = new (buf) S(args);  // конструктор S вызывается в buf
```

**Зачем**: `std::vector` (буфер заранее, конструирование по мере добавления), memory pools.

---

## Q47. Как корректно удалить объект из placement new

1. Явно вызвать деструктор: `ps->~S();`
2. Освободить исходную память: `delete[] buf;`

Делать `delete ps` — **UB**: память была выделена через `new[]`, а не `new`.

---

## Q48. Undefined, Unspecified, Implementation-defined behavior

- **Undefined behavior (UB)**: стандарт не накладывает никаких требований. Компилятор волен делать что угодно. Классика: разыменование nullptr, выход за пределы массива, signed overflow.
- **Unspecified behavior**: поведение не определено стандартом, но не UB. Одно из допустимых. Пример: порядок вычисления аргументов функции.
- **Implementation-defined**: поведение на усмотрение реализации, но обязана задокументировать. Пример: размер `int`, знаковость `char`.

---

## Q49. Наблюдаемое поведение программы (observable behaviour)

Всё, что компилятор обязан сохранить при оптимизациях:
- Операции с `volatile` объектами (в порядке кода).
- Ввод/вывод через стандартные потоки.
- Взаимодействие с ОС и внешним миром.

Зачем нужно: компилятор может переставлять, удалять, добавлять любые инструкции — **пока наблюдаемое поведение не меняется**. Это легальная основа для большинства оптимизаций (as-if rule).

---

## Q50. `this`

Указатель на текущий объект внутри нестатического метода. Тип: `T*` (или `const T*` в const-методе).

**Можно ли присваивать в `this`?** — Нет. `this` — rvalue (не переменная), нельзя написать `this = &other`.

---

## Q51. Можно ли `delete this`?

**Технически можно**, если:
- Объект создан через `new` (не стек, не static).
- После `delete this` не обращаться ни к каким членам объекта.

На практике — антипаттерн. Используется в некоторых COM-объектах.

---

## Q52. Явное/неявное преобразование типов

- **Неявное**: происходит автоматически. `int → double`, конструктор без `explicit`.
- **Явное**: программист указывает явно. C-style: `(int)x`. C++-style: `static_cast<int>(x)`.

---

## Q53. Варианты явного приведения типов в C++

- **`static_cast<T>`**: проверка на этапе компиляции. Для связанных типов, числовых конверсий, upcast/downcast (без runtime-проверки).
- **`dynamic_cast<T>`**: runtime-проверка. Только для указателей/ссылок на полиморфные типы. Downcast с проверкой.
- **`const_cast<T>`**: убирает/добавляет `const`. Только это — других преобразований не делает.
- **`reinterpret_cast<T>`**: бинарная переинтерпретация. Почти всегда UB кроме узких разрешённых случаев.
- **C-style `(T)x`**: пробует static, const, reinterpret в порядке — непредсказуемо. Не использовать.

## Q54. Макросы vs функции

**Макрос** — текстовая подстановка препроцессора, до компиляции.

| | Макрос | Функция |
|---|---|---|
| Проверка типов | ❌ | ✅ |
| Область видимости | ❌ | ✅ |
| Побочные эффекты в аргументах | Каждый аргумент вычисляется при каждом упоминании в теле | Аргументы вычислены один раз |
| Отладка | Трудно | Легко |

`MAX(++x, --y)` → `((++x) > (--y) ? (++x) : (--y))` — x/y меняются дважды.

## Q55. Что делает препроцессор

- `#include` — подставляет содержимое файла
- `#define`/`#undef` — определяет/удаляет макросы
- `#if`/`#ifdef`/`#ifndef`/`#elif`/`#else`/`#endif` — условная компиляция
- `#pragma` — зависит от компилятора (once, pack, optimize...)
- `#error`/`#warning` — остановить или предупредить

Работает **до** компиляции, не знает о типах и синтаксисе C++.

---

## Q56. ODR — One Definition Rule

Во всей программе — ровно одно **определение** каждой сущности с внешним связыванием. Нарушение — UB (линкер может не поймать).

**Исключения**: `inline`-функции, шаблоны, `constexpr`/`consteval`-функции — могут быть определены в нескольких единицах трансляции (при условии идентичности).

---

## Q57. Когда можно размещать определение функции в header

Можно:
- `inline`-функции
- `constexpr`/`consteval`-функции (неявно inline)
- Шаблоны (неявно inline)
- Методы класса, определённые прямо в теле класса (неявно inline)

Нельзя (нарушение ODR):
- Обычные функции (не inline, не шаблон)
- Переменные без `inline`/`constexpr`

## Q58. Forward declaration

Объявление типа без определения. Создаёт **неполный тип**.

```cpp
class C;      // forward declaration
enum class E; // OK: базовый тип int известен
```

**Что можно с неполным типом**: указатель `C*`, ссылка `C&`, объявления функций с `C*`.  
**Что нельзя**: создать переменную, нестатическое поле, `sizeof`, наследоваться.

---

## Q59. Время жизни объекта `void f(X a) {}`

- **Начало**: при вызове `f` — конструктор копирования (или перемещения) из аргумента.
- **Конец**: при завершении `f` — деструктор `a`.

---

## Q60. Время жизни `auto pa = new X`

- **Начало**: при выполнении `new X` — конструктор вызван, память выделена.
- **Конец**: при `delete pa` — деструктор, затем освобождение памяти.
- Если `delete` не вызван — **утечка памяти**.

---

## Q61. Порядок создания/разрушения `void f() { X a; { Y b; } }`

- Создание: `a`, затем `b` (когда входим во вложенный блок).
- Разрушение: `b` (при выходе из вложенного блока), затем `a` (при выходе из функции).

---

## Q62. Подобъекты `struct X : Y, Z { W w; V v; }`

**Порядок создания**: Y → Z → w → v (базовые в порядке наследования, затем поля в порядке объявления).  
**Порядок разрушения**: v → w → Z → Y (строго обратный).

---

## Q63. Временный объект и его время жизни

Временный объект (prvalue) живёт до конца **полного выражения** (выражения-оператора), если не привязан к ссылке.

**Привязка к `const T&` или `T&&`** — продлевает время жизни до конца области видимости ссылки:
```cpp
const T& r = T();  // OK: T() живёт пока жива r
T&& rr = T();      // аналогично
```

**Исключения**: не продлевается при возврате из функции через ссылку, при привязке к параметру функции.

---

## Q64. Инициализация по умолчанию: базовые vs пользовательские типы

- **Базовые типы** (`int`, `double`, `bool`): **не инициализируются** при default-init — значение неопределено (UB при чтении).
- **Пользовательские типы (классы)**: вызывается конструктор по умолчанию.
- **Value-initialization** `T{}` или `T()`: базовые обнуляются, классы — конструктор.

```cpp
int x;      // UB (garbage)
int x{};    // 0
MyClass obj; // default constructor
```

## Q65. `push_back` vs `emplace_back`

| | `push_back(obj)` | `emplace_back(args...)` |
|---|---|---|
| Принимает | готовый объект | аргументы конструктора |
| Что происходит | copy/move в вектор | конструирование на месте |
| Лишние операции | иногда есть | нет |

```cpp
v.push_back(T(42));   // создаёт T(42), потом перемещает
v.emplace_back(42);   // T конструируется прямо в памяти вектора
```

**Предпочитайте `emplace_back`**.

## Q66. Ссылки на какие объекты уместно возвращать

Только на объекты, время жизни которых **превышает время жизни ссылки**:
- Статические переменные
- Поля класса (если возвращаем из метода)
- Параметры функции (если переданы по ссылке и переживут вызов)

**Никогда** — на локальные переменные. Это dangling reference → UB.

## Q67. `std::move`

`static_cast<T&&>(x)` — приводит lvalue к xvalue (rvalue-ссылке).  
Само по себе ничего не перемещает. Разрешает вызов move-конструктора/move-оператора.

После `std::move(x)` — `x` в валидном, но неопределённом состоянии.

---

## Q68. `std::forward` *

Используется в шаблонах для "совершенной передачи" (perfect forwarding):
```cpp
template<typename T>
void wrapper(T&& arg) {
    f(std::forward<T>(arg));  // сохраняет lvalue/rvalue категорию
}
```
Без `forward` — `arg` всегда lvalue внутри функции (у него есть имя).

---

## Q69. `T f(int n) { return n; } auto x = f(101);`

Один конструктор `T(int)` при создании объекта в `f`.  
**Copy elision** (гарантирована с C++17) — `x` создаётся сразу на своём месте, copy/move конструкторы не вызываются.

---

## Q70. `T f(T x) { return x; } auto y = f(101);`

1. `T(int)` — неявная конверсия 101 → T при передаче в параметр.
2. При возврате `x`: NRVO **может** устранить copy/move, но не гарантировано (x — параметр).  
   Без NRVO: `T(T&&)` (move) или `T(const T&)` (copy).

---

## Q71. `T x; x = T();`

1. `T()` — default constructor для `x`.
2. `T()` — default constructor для временного `T()`.
3. `x = T()` — **move assignment** (T() — rvalue).
4. Деструктор временного.

---

## Q72. Корректно ли `f(x.get_value(), std::move(x))`?

**Нет, некорректно**. Порядок вычисления аргументов — unspecified. Если `std::move(x)` вычислится первым — `x.get_value()` работает с moved-from объектом. Поведение зависит от компилятора.

---

## Q73. `void f(std::vector<T>& v) { v.resize(100); }`

При `resize` от меньшего к большему:
- Если `capacity` достаточно: `T()` вызывается для каждого нового элемента (100 - текущий size раз).
- Если нет — **реаллокация**: move (или copy если нет noexcept move) всех существующих элементов, затем default-construct новых.

Итого: 0-100 вызовов `T()` + потенциально move/copy существующих при реаллокации.

---

## Q74. `class X : public Y { public: void f(); }` если Y уже имеет `void f()`

**Hiding (сокрытие)**, а не переопределение. `X::f()` скрывает `Y::f()` — в области видимости X видна только `X::f`. Обе сосуществуют.

```cpp
X x;
x.f();                          // X::f
static_cast<Y&>(x).f();         // Y::f
```

Если `Y::f()` виртуальная — это **override**. Если нет — просто hiding.
